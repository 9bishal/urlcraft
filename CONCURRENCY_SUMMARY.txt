┌─────────────────────────────────────────────────────────────────────────────────┐
│ 1️⃣  NODE.JS EVENT-DRIVEN ARCHITECTURE                                          │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  How it works:                                                                   │
│  ├─ Single-threaded JavaScript engine (Event Loop)                             │
│  ├─ Handles 1000s of concurrent connections                                   │
│  ├─ Non-blocking I/O operations                                               │
│  ├─ Async/Await for database queries                                          │
│  └─ Memory efficient (~25MB base)                                             │
│                                                                                  │
│  Traditional Blocking:                                                          │
│  ├─ Request 1 → [DB Block] → Response                                         │
│  ├─ Request 2 → [DB Block] → Response (waits for Req 1)                       │
│  └─ ❌ Only 1 request at a time per thread                                     │
│                                                                                  │
│  Node.js Non-Blocking:                                                          │
│  ├─ Request 1 → [DB Query] ─┐                                                 │
│  ├─ Request 2 → [DB Query] ─┼─ Event Loop ─ Responses when ready             │
│  ├─ Request 3 → [DB Query] ─┘                                                 │
│  └─ ✅ 1000s of requests handled concurrently                                  │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 2️⃣  POSTGRESQL CONNECTION POOLING                                              │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  Configuration:                                                                  │
│  ├─ Pool Size: 20 connections                                                  │
│  ├─ Max Connections: Reuses existing connections                               │
│  └─ Connection Timeout: 2000ms                                                 │
│                                                                                  │
│  Without Pooling:                                                               │
│  ├─ Request 1 → CREATE DB Connection → Query → CLOSE                          │
│  ├─ Request 2 → CREATE DB Connection → Query → CLOSE                          │
│  └─ ❌ Each connection takes ~100ms to create                                  │
│                                                                                  │
│  With Pooling:                                                                  │
│  ├─ Pool: [Conn 1] [Conn 2] ... [Conn 20]                                     │
│  ├─ Request 1 → Grab Conn 1 → Query → Return Conn 1                           │
│  ├─ Request 2 → Grab Conn 2 → Query → Return Conn 2                           │
│  ├─ Request 3 → Grab Conn 1 (reused) → Query → Return Conn 1                  │
│  └─ ✅ Handles 100+ concurrent requests with 20 connections                   │
│                                                                                  │
│  Key Benefit: Reusing connections eliminates connection creation overhead      │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 3️⃣  REDIS IN-MEMORY CACHE                                                      │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  Caching Strategy:                                                               │
│  ├─ Cache popular URLs in Redis                                                │
│  ├─ Cache URL click counts                                                     │
│  ├─ Cache refresh tokens                                                       │
│  └─ Auto-expire cache entries                                                  │
│                                                                                  │
│  Performance Impact:                                                             │
│  ├─ Cache MISS: ~50ms (query DB + cache result)                               │
│  ├─ Cache HIT: ~1ms (read from Redis)                                         │
│  ├─ 50x FASTER for cached data!                                               │
│  └─ Typical Hit Rate: 90-95% for popular URLs                                 │
│                                                                                  │
│  Example Scenario (1000 users accessing same URL):                              │
│  ├─ User 1 → Cache MISS → Query DB → Store in Redis → Response (50ms)         │
│  ├─ User 2 → Cache HIT → Read Redis → Response (1ms)                          │
│  ├─ User 3 → Cache HIT → Read Redis → Response (1ms)                          │
│  ├─ ...                                                                         │
│  ├─ User 1000 → Cache HIT → Read Redis → Response (1ms)                       │
│  └─ ✅ DB hit just once, then 999 users get responses from cache              │
│                                                                                  │
│  Scalability: Can handle 10,000+ requests/second for popular URLs             │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 4️⃣  RATE LIMITING (DDoS PROTECTION)                                           │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  Configuration:                                                                  │
│  ├─ Window: 15 minutes                                                         │
│  ├─ Max Requests: 30 per IP per window                                        │
│  ├─ Per-IP Tracking: Prevents single user from overwhelming                   │
│  └─ Automatic Blocking: Returns 429 Too Many Requests                        │
│                                                                                  │
│  Protection Levels:                                                              │
│  ├─ User A (IP 192.168.1.1) → Requests 1-30 ✅ Allowed                        │
│  ├─ User A (IP 192.168.1.1) → Request 31 ❌ BLOCKED (rate limited)            │
│  ├─ User B (IP 192.168.1.2) → Requests 1-30 ✅ Allowed (separate limit)       │
│  └─ After 15 min → User A counter resets ✅ Can make requests again           │
│                                                                                  │
│  Key Benefit: Fair resource distribution, prevents DDoS attacks               │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────────┐
│ 5️⃣  JWT STATELESS AUTHENTICATION                                              │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  Why Stateless?                                                                  │
│  ├─ Each server independently verifies tokens (no shared state)                │
│  ├─ No session synchronization needed across servers                           │
│  ├─ Perfect for horizontal scaling                                             │
│  └─ Reduces database queries for auth checks                                   │
│                                                                                  │
│  Stateful (Traditional):                                                        │
│  ├─ Request → Server A → Check session store → Response                       │
│  ├─ Request → Server B → Check session store → Response                       │
│  └─ ❌ Session data must be synced across servers                             │
│                                                                                  │
│  Stateless (JWT):                                                               │
│  ├─ Request + JWT → Server A → Verify (locally) → Response                    │
│  ├─ Request + JWT → Server B → Verify (locally) → Response                    │
│  └─ ✅ No sync needed, servers are independent                                 │
│                                                                                  │
│  Key Benefit: Enables true horizontal scaling                                   │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘
